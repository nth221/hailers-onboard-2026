# House Prices 발표 대본

예상 발표 시간: 약 10~12분  
발표자: 최옥토

> 대본을 전부 외우기보다 각 슬라이드의 첫 문장과 마지막 문장만 익히는 것을 권장합니다.  
> 괄호 안의 내용은 발표자가 참고하는 표시이며, 실제로 읽지 않아도 됩니다.

---

## 1. 표지

안녕하세요. House Pricing 프로젝트를 진행한 최옥토입니다.  
이번 프로젝트에서는 주택의 여러 특성을 이용해 판매 가격을 예측하고, 어떤 실험이 실제 성능 개선으로 이어졌는지 확인했습니다.

## 2. 목차

발표는 먼저 문제와 데이터를 간단히 설명한 뒤, 전처리와 모델 학습 과정을 말씀드리겠습니다.  
그다음 실험별 점수 변화와 잘되지 않았던 시도까지 살펴보고, 마지막으로 프로젝트에서 얻은 결론을 정리하겠습니다.

## 3. 문제 정의와 목표

이 문제의 입력은 위치, 면적, 품질, 연식 등 주택을 설명하는 79개 특성이고, 출력은 실제 판매 가격인 `SalePrice`입니다.  
정해진 클래스 중 하나를 고르는 문제가 아니라 연속된 가격을 예측하므로 회귀 문제입니다.

평가는 RMSLE를 사용합니다. 쉽게 말하면 비싼 집에서 발생한 큰 금액 차이보다, 실제 가격에 비해 몇 퍼센트 정도 틀렸는지를 더 중요하게 보는 지표입니다.  
과제에서 요구한 직접 구현 MLP를 완성하는 것과 함께, Public 점수 0.1233 미만으로 1,000위 안에 들어가는 것을 목표로 잡았습니다.

## 4. 학습 데이터와 테스트 데이터

학습 데이터는 1,460개 주택과 정답 가격을 포함하고 있고, 테스트 데이터는 1,459개 주택으로 구성되어 있습니다.  
모델은 train에서 주택 특성과 가격의 관계를 학습하고, test의 각 `Id`에 해당하는 `SalePrice`를 예측합니다.

최종 제출 파일은 복잡하지 않습니다. 테스트 데이터의 `Id` 순서를 유지한 상태에서, 예측한 `SalePrice`를 한 줄씩 넣으면 됩니다.

## 5. 79개 변수를 7개의 가격 신호로 이해

처음에는 컬럼이 79개라서 각각을 따로 보면 관계를 이해하기 어려웠습니다.  
그래서 입지, 규모, 품질, 연식, 공간 구성, 부대시설, 거래 맥락이라는 7개 그룹으로 묶어 살펴봤습니다.

예를 들어 `Neighborhood`는 지역의 기본 가격대를, `GrLivArea`는 실제 생활 공간을, `OverallQual`은 집의 전반적인 완성도를 설명합니다.  
이렇게 묶으니 각 컬럼이 가격에 어떤 정보를 주는지 훨씬 명확해졌습니다.

## 6. 타깃 분포와 log1p

왼쪽 원본 가격 분포를 보면 일부 고가 주택 때문에 오른쪽 꼬리가 길게 나타납니다.  
이 상태에서는 값이 큰 몇 개의 주택이 학습에 과도한 영향을 줄 수 있습니다.

그래서 `SalePrice`에 `log1p`를 적용했습니다. 변환 후에는 분포가 중앙에 더 안정적으로 모이고, 모델도 절대 금액보다 가격 비율의 차이를 학습하게 됩니다.  
회귀 문제라 클래스 불균형은 없지만, 이런 타깃의 왜도는 따로 관리할 필요가 있었습니다.

## 7. 가격과 관련이 큰 변수

가격과 가장 강하게 연결된 변수는 `OverallQual`과 `GrLivArea`였습니다.  
즉 집의 전반적인 품질이 좋고 실제 생활 면적이 넓을수록 가격이 올라가는 경향이 뚜렷했습니다.

그다음으로 차고 크기, 지하실 면적, 1층 면적, 욕실 수, 건축 연도 등이 가격 차이를 보완했습니다.  
참고한 외부 EDA의 변수 중요도도 비슷한 방향을 보여서, 이 관계를 주요 실험 가설의 근거로 사용했습니다.

## 8. Neighborhood의 역할

`Neighborhood`는 단순한 주소 정보라기보다 집값의 출발점을 정하는 변수에 가까웠습니다.  
같은 크기와 품질의 집이라도 어느 지역에 있는지에 따라 기본 가격대가 달라질 수 있기 때문입니다.

실제로 지역별 중앙가격 차이가 크게 나타났기 때문에 이 변수를 제거하지 않았고, One-Hot Encoding으로 각 지역 정보를 보존했습니다.

## 9. 결측치 처리

이 데이터에서 결측치는 모두 오류를 의미하지 않습니다.  
예를 들어 `PoolQC`가 비어 있는 경우는 수영장 품질을 조사하지 못했다기보다, 수영장 자체가 없는 경우가 대부분입니다.

따라서 수치형 결측치는 극단값의 영향을 덜 받는 중앙값으로 채웠고, 범주형 결측치는 `Missing`이라는 별도 범주로 처리했습니다.  
이렇게 하면 시설이 없다는 정보도 모델에 남길 수 있습니다. 중복 데이터는 확인 결과 없었습니다.

## 10. 이상치 실험

외부 EDA에서는 생활 면적이 4,000을 넘는 일부 주택을 이상치로 보고 제거하는 방법이 제안되어 있었습니다.  
저도 일반화 성능이 좋아질 것이라고 생각해 같은 실험을 했지만, 실제 Public 점수는 오히려 나빠졌습니다.

테스트 데이터에도 큰 주택이 존재할 수 있는데, 필요한 사례까지 학습에서 지웠을 가능성이 있다고 판단했습니다.  
그래서 이상치 제거는 원복했습니다. 이 실험을 통해 정석적인 방법도 현재 검증 결과를 보고 채택해야 한다는 점을 확인했습니다.

## 11. 전처리 파이프라인

전처리는 결측치 처리, One-Hot Encoding, Standard Scaling을 거쳐 PyTorch Tensor로 변환하는 순서입니다.  
여기서 가장 중요하게 지킨 원칙은 전처리 기준을 각 fold의 학습 데이터에서만 계산하는 것입니다.

학습 데이터에서 구한 중앙값과 평균, 표준편차, 범주 목록을 validation과 test에 그대로 적용했습니다.  
validation을 전처리 학습에 사용하면 정답을 미리 본 것과 비슷한 데이터 누수가 발생해 OOF가 실제보다 좋아 보일 수 있기 때문입니다.

## 12. Feature Engineering

Feature Engineering에서는 컬럼 수를 무작정 늘리기보다, 흩어진 정보를 한 번에 보여주는 데 집중했습니다.  
`HouseAge`는 판매 연도에서 건축 연도를 빼서 판매 당시의 실제 연식을 나타냅니다.

`TotalBath`는 전체 욕실과 반 욕실, 지하실 욕실을 사용 가치에 맞춰 하나의 수치로 합친 변수입니다.  
반면 `TotalSF`처럼 기존 면적 변수와 정보가 많이 겹치는 feature는 성능이 나빠져 최종 버전에서 제외했습니다.

## 13. 직접 구현한 MLP

과제의 필수 모델은 `torch.nn.Module`을 이용해 직접 구현한 MLP입니다.  
입력층 다음에 128개와 64개의 노드를 가진 두 hidden layer를 배치했고, ReLU와 Dropout 0.1을 적용했습니다.

손실 함수는 로그 타깃에 대한 MSE, optimizer는 AdamW를 사용했습니다.  
학습과 validation loop를 직접 작성했고, validation 성능이 가장 좋았던 checkpoint를 저장한 뒤 patience 20의 early stopping을 적용했습니다.

## 14. TabM 확장 실험

필수 MLP를 완성한 뒤, 같은 계열의 모델을 더 안정적으로 확장할 방법으로 TabM을 실험했습니다.  
TabM은 여러 개의 독립 MLP를 각각 학습하는 대신, 대부분의 가중치를 공유하면서 여러 관점의 예측을 함께 학습하는 구조입니다.

이 방식으로 단일 MLP의 OOF는 0.13614에서 0.12370으로 낮아졌고, Public 점수도 0.12314에서 0.12139로 개선됐습니다.  
즉 완전히 다른 파이프라인으로 바꾼 것이 아니라, 기존 MLP의 장점을 유지하면서 앙상블 효과를 모델 내부에 가져온 실험입니다.

## 15. 5-Fold와 타깃 중심화

데이터가 1,460개로 많지 않기 때문에 한 번의 train-validation 분할만으로 모델을 판단하면 결과가 우연에 크게 흔들릴 수 있습니다.  
그래서 데이터를 다섯 부분으로 나누고, 매번 네 부분으로 학습하고 남은 한 부분으로 검증하는 5-Fold를 사용했습니다.

각 fold에서는 로그 가격에서 해당 fold의 train 평균을 빼서 0 주변의 차이를 학습하도록 했습니다. 이것이 타깃 중심화입니다.  
예측할 때는 평균을 다시 더하고 `expm1`으로 실제 가격을 복원했으며, 다섯 모델의 test 예측을 평균해 제출값을 만들었습니다.

## 16. 모델별 비교

모델마다 표 데이터를 처리하는 방식과 장단점이 달랐습니다.  
기본 MLP는 구조와 학습 과정을 직접 설명하기 좋지만, 작은 데이터와 희귀 사례에 민감했습니다.

Huber Loss는 극단 오차의 영향을 줄여주지만 이번 평가 지표와는 잘 맞지 않았고, FT-Transformer는 변수 간 상호작용을 표현할 수 있지만 적은 데이터와 제한된 튜닝에서 불안정했습니다.  
CatBoost는 작은 표 데이터에 강한 기준 모델이었지만 과제 정책상 최종 모델이 아닌 비교용 baseline으로만 사용했습니다. 최종적으로 OOF와 Public을 모두 개선한 모델은 TabM이었습니다.

## 17. 실험 성능 변화

성능은 한 번에 큰 모델로 교체해서 올라간 것이 아니라, 한 가지씩 바꾸며 개선을 누적한 결과입니다.  
기본 MLP의 0.16031에서 Dropout으로 0.15185, 연식과 욕실 feature를 추가해 0.14745까지 개선했습니다.

이후 타깃 중심화와 5-Fold를 적용한 MLP가 0.12314를 기록했고, 마지막 TabM 실험에서 0.12139까지 낮췄습니다.  
실험할 때는 한 번에 하나만 바꾸고, 좋아지면 유지하고 나빠지면 원복하는 원칙을 지켰습니다.

## 18. 최종 Public 결과

최종 TabM의 Public RMSLE는 0.12139였고, 최종 순위는 619위였습니다.  
기존 최고 MLP보다 점수는 0.00175 낮아졌고, 순위는 959위에서 619위로 340계단 상승했습니다.

Required 목표였던 0.1233 미만과 1,000위 이내를 달성했고, Ambitious 목표인 0.1199와의 차이는 0.00149까지 줄였습니다.

## 19. OOF와 Public의 관계

Public 점수만 좋아졌다면 leaderboard 데이터에 우연히 잘 맞았을 가능성도 생각해야 합니다.  
하지만 TabM은 로컬 OOF도 0.13614에서 0.12370으로 약 9.1% 개선됐습니다.

OOF와 Public이 같은 방향으로 좋아졌기 때문에, 이번 개선은 단순한 제출 운보다는 모델의 일반화 성능이 실제로 좋아진 결과라고 판단했습니다.

## 20. TabM이 효과적이었던 이유

이번 결과에서 모델을 무조건 복잡하게 만드는 것이 항상 답은 아니었습니다.  
FT-Transformer보다 오히려 기존 MLP 구조를 확장한 TabM이 더 좋은 결과를 냈습니다.

전처리, feature, 타깃 중심화, 5-Fold처럼 이미 검증한 MLP 파이프라인을 그대로 유지하면서 여러 예측을 공동 학습한 점이 효과적이었다고 해석했습니다.  
즉 이번 데이터에서는 새로운 표현 방식을 처음부터 다시 학습하는 것보다, 안정된 MLP의 분산을 줄이는 방식이 더 잘 맞았습니다.

## 21. 잘되지 않았던 실험

성능이 나빠진 실험도 최종 방향을 정하는 데 도움이 됐습니다.  
이상치 제거는 test에 필요한 큰 주택 사례까지 없앴을 가능성이 있었고, `TotalSF`는 기존 면적 변수와 정보가 겹쳐 노이즈가 늘어난 것으로 보입니다.

Huber Loss는 이상치에 강하지만 Kaggle의 RMSLE와 최적화 방향이 완전히 같지는 않았고, FT-Transformer는 데이터 수에 비해 충분한 튜닝을 하지 못했습니다.  
따라서 모델의 일반적인 장점만 보고 결과를 예상하기보다, 같은 전처리와 검증 기준에서 직접 비교하는 것이 중요했습니다.

## 22. 재현성과 협업

실험 결과를 다시 만들 수 있도록 seed는 42로 고정했고, 5-Fold 분할과 fold별 전처리 규칙을 동일하게 유지했습니다.  
모든 모델은 best validation checkpoint를 기준으로 저장하고, 다섯 개의 test 예측을 평균하는 같은 절차를 따랐습니다.

팀에서는 모델별 실험을 나누되 같은 데이터와 평가 기준을 공유했고, 좋은 결과뿐 아니라 실패한 가설과 원인도 함께 기록해 다음 실험을 정했습니다.

## 23. 결론

이번 프로젝트에서 가장 중요하게 배운 점은 좋은 모델 하나를 바로 찾는 것보다, 데이터를 이해하고 검증 가능한 개선을 쌓는 과정이 중요하다는 것입니다.  
가격의 왜도와 결측치 의미를 먼저 확인하고, 데이터 누수가 없는 전처리와 5-Fold 검증을 만든 것이 성능 개선의 기반이 됐습니다.

그 기반 위에서 직접 구현한 MLP를 TabM으로 확장했고, 최종적으로 Public 0.12139와 619위를 기록했습니다.  
시간이 더 주어진다면 TabM의 세부 하이퍼파라미터와 모델 간 OOF 기반 앙상블을 추가로 실험해 0.1199에 도전해 보겠습니다. 감사합니다.

## 예상 질문에 대한 짧은 답변

### Q1. 왜 SalePrice에 log1p를 적용했나요?

고가 주택 때문에 생긴 긴 꼬리를 줄이고, 절대 금액보다 가격 비율의 오차를 학습하기 위해 적용했습니다. Kaggle의 RMSLE 평가 방식과도 잘 맞습니다.

### Q2. 왜 이상치를 제거하지 않았나요?

제거하면 일반화가 좋아질 것이라는 가설을 세웠지만 실제 Public 점수가 나빠졌습니다. test에도 큰 주택이 존재할 수 있어 필요한 학습 사례까지 잃었다고 판단해 원복했습니다.

### Q3. 타깃 중심화는 데이터 누수가 아닌가요?

각 fold의 train 타깃 평균만 사용하고 validation이나 test의 정답은 사용하지 않았기 때문에 누수가 아닙니다. 예측 시에는 해당 train 평균만 다시 더했습니다.

### Q4. CatBoost가 있는데 왜 최종 모델로 사용하지 않았나요?

CatBoost는 강한 비교 기준으로 사용했지만, 과제에서는 직접 정의한 PyTorch 신경망이 필수였고 tree ensemble은 최종 모델로 인정되지 않았기 때문입니다.

### Q5. TabM이 과제 정책에 맞나요?

과제의 필수 구현 조건은 직접 작성한 MLP로 먼저 충족했습니다. TabM은 그 결과와 비교하기 위한 추가 확장 실험으로 구분해 사용했습니다.

### Q6. OOF와 Public 점수가 왜 다른가요?

OOF는 train 내부의 다섯 validation 구간을 대상으로 계산하고, Public은 정답을 모르는 Kaggle test 일부를 대상으로 계산하기 때문입니다. 데이터 구성과 어려운 사례의 비율이 달라 절댓값은 달라질 수 있으며, 개선 방향을 함께 보는 것이 중요합니다.
